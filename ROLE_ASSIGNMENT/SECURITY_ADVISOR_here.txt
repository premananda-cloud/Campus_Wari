PHASE 1: UNDERSTAND THE AUTH FLOW
===========================================================

STEP 1.1 - The Signup Flow
  1. User sends email + name + password to /auth/signup
  2. Server checks: is email already registered?
  3. Server checks: is password strong enough? (minimum length, etc.)
  4. Server hashes the password (NEVER stored as plain text)
  5. Server stores user in database with the HASH
  6. Server returns success (optionally, auto-login with token)

  Question you must answer:
    - Do we verify college email? (send verification link? check domain?)
    - What password rules do we enforce?

STEP 1.2 - The Login Flow
  1. User sends email + password to /auth/login
  2. Server finds user by email in database
  3. Server compares provided password with stored hash
  4. If match: server creates a JWT token with user ID and expiry
  5. Server returns the token to the frontend
  6. Frontend stores the token (localStorage or httpOnly cookie)
  7. Frontend sends the token with every subsequent request

  Questions you must answer:
    - How long should a token last? (15 minutes? 1 hour? 1 day?)
    - Where does the frontend store the token?
    - What happens when the token expires?

STEP 1.3 - The Protected Route Flow
  1. User requests /users/me or /listings/create
  2. Frontend includes token in Authorization header:
       Authorization: Bearer <token>
  3. Backend middleware extracts and verifies the token
  4. If valid: request proceeds with user identity known
  5. If invalid/expired: return 401 Unauthorized

PHASE 2: SECURITY MODELING - THINK LIKE AN ATTACKER
===========================================================

STEP 2.1 - Threat Scenarios To Consider

  Scenario A: Token Theft
    Someone copies a user's JWT token.
    Can they impersonate that user forever?
    -> Mitigation: Short token expiry. Refresh tokens.

  Scenario B: Brute Force
    Someone tries thousands of password combinations.
    Can they guess a weak password?
    -> Mitigation: Rate limiting. Minimum password strength.

  Scenario C: Password Leak
    The database is somehow exposed.
    Are user passwords readable?
    -> Mitigation: Passwords are HASHED. Hashing is one-way.

  Scenario D: Lost Password
    A genuine user forgets their password.
    Are they locked out forever?
    -> Mitigation: Password reset flow (email-based).

  Scenario E: Unauthorized Access
    User A tries to view User B's private messages.
    Can they?
    -> Mitigation: Every endpoint checks ownership.

STEP 2.2 - Design Your Contingency Plan

  For each scenario above, write down:
    - What's the worst that could happen?
    - How do we prevent it?
    - If prevention fails, how do we detect it?
    - If detection happens, how do we recover?

  Example for "Lost Password":
    Worst case: User locked out permanently.
    Prevention: Have a password reset flow from day one.
    Detection: User reports it (or we see failed login spikes).
    Recovery: Reset link sent to registered email. New password set.

PHASE 3: IMPLEMENTATION PLAN
===========================================================

CHECKPOINT 1 - Password Hashing Utility
  Write a utility function (or understand the library) that:
    - Takes a plain text password
    - Returns a hash (with built-in salt)
    - Can later verify: does this plain text match this hash?

  This is TWO functions: hash_password() and verify_password()

CHECKPOINT 2 - JWT Token Utility
  Write utility functions that:
    - Create a token with user ID and expiry time
    - Verify a token and extract the user ID
    - Handle expired tokens gracefully

  This is TWO functions: create_access_token() and decode_token()

CHECKPOINT 3 - Signup Endpoint Integration
  Work with Backend person so /auth/signup:
    - Uses YOUR hash_password() function
    - Stores the hash, never the plain text
    - Returns appropriate success/error

CHECKPOINT 4 - Login Endpoint Integration
  Work with Backend person so /auth/login:
    - Uses YOUR verify_password() function
    - Uses YOUR create_access_token() function
    - Returns the token with user info

CHECKPOINT 5 - Protected Route Middleware
  Create a dependency that Backend can use on any route:
    from utils.auth import get_current_user

    @router.get("/users/me")
    def get_me(current_user = Depends(get_current_user)):
        return current_user

  This middleware:
    - Extracts token from Authorization header
    - Decodes and verifies it
    - Fetches the user from database
    - Returns the user object (or raises 401)

CHECKPOINT 6 - Password Reset Flow (Phase 2 feature)
  Design (and later implement):
    - "Forgot password?" link on login page
    - User enters email
    - If email exists, send reset link (with expiring token)
    - User clicks link, enters new password
    - Password is hashed and updated

===========================================================
TIPS FROM SOMEONE WHO'S DONE THIS
===========================================================

1. NEVER ROLL YOUR OWN CRYPTO
   Use well-tested libraries: bcrypt for hashing, PyJWT for tokens.
   Never write your own hashing algorithm. Just don't.

2. TOKEN EXPIRY IS YOUR FRIEND
   A token that lasts forever is a stolen token that works forever.
   Start with 30-minute expiry. You can always extend it later.

3. THE FRONTEND NEEDS TO HANDLE 401
   When a token expires, the API returns 401.
   The frontend should catch this and redirect to login.
   Coordinate this with the Frontend person.

4. BE PARANOID, BUT PRACTICAL
   Yes, httpOnly cookies are more secure than localStorage.
   But for a campus platform? localStorage with good expiry is fine.
   Don't let perfect be the enemy of shipped.

5. DOCUMENT YOUR SECURITY DECISIONS
   Write down: "We chose 30-min token expiry because X."
   Future you (and future teammates) will thank you.

6. WORK WITH BACKEND, NOT AGAINST THEM
   You provide the utility functions.
   Backend calls them in the endpoints.
   This separation keeps things clean and testable.
