PHASE 1: SET UP POSTGRESQL
===========================================================

STEP 1.1 - Install PostgreSQL
  On Debian/Ubuntu:
    sudo apt update
    sudo apt install postgresql postgresql-contrib

  On Mac:
    brew install postgresql

  On Windows:
    Download from postgresql.org

STEP 1.2 - Log In As Admin And Create Your User
  sudo -u postgres psql

  Inside the psql terminal:
    CREATE USER your_name WITH SUPERUSER PASSWORD 'choose_a_strong_password';
    \q

STEP 1.3 - Log In As Yourself And Create The Database
  psql -U your_name -d postgres -h localhost -W
  (The -W flag makes it prompt for your password)

  Inside psql:
    CREATE DATABASE campus_wari_db;
    \c campus_wari_db
    (Now you are connected to the new database)

===========================================================
PHASE 2: DOCUMENT YOUR CREDENTIALS
===========================================================

STEP 2.1 - Create .env File
  In the project root, create a file called .env:

    DB_USER=your_name
    DB_PASSWORD=your_strong_password
    DB_HOST=localhost
    DB_PORT=5432
    DB_NAME=campus_wari_db
    DATABASE_URL=postgresql://your_name:your_password@localhost:5432/campus_wari_db

  WARNING: .env contains secrets. It goes in .gitignore IMMEDIATELY.
  Never commit it. Share credentials through a private channel.

STEP 2.2 - Create .env.example (Safe to Commit)
  Same keys, no real values:

    DB_USER=your_db_user
    DB_PASSWORD=your_db_password
    DB_HOST=localhost
    DB_PORT=5432
    DB_NAME=campus_wari_db
    DATABASE_URL=postgresql://user:password@localhost:5432/dbname

  This helps teammates understand what variables they need to set.

PHASE 3: DESIGN THE SCHEMA (THINK BEFORE YOU CREATE)
===========================================================

STEP 3.1 - Ask These Questions First
  - Who are our users? What defines them?
    -> email, name, department, year, password hash

  - What do users create/post?
    -> listings for tutoring, notes sharing, collaborations, hangouts

  - How do users interact?
    -> they message each other about listings

  - What statuses can a listing have?
    -> active, fulfilled, expired

STEP 3.2 - Define Tables And Relationships

  USERS table:
    - id (unique identifier, auto-incrementing)
    - email (unique, must be college email ideally)
    - name (their real name or display name)
    - department (which department they belong to)
    - year (which year of study)
    - password_hash (NEVER store plain passwords)
    - created_at (when they joined)

  LISTINGS table:
    - id (unique identifier)
    - user_id (who posted it, links to users table)
    - type (TUTORING, NOTES, COLLAB, or HANGOUT)
    - title (short headline)
    - description (details)
    - subject (what subject/course this relates to)
    - status (ACTIVE, FULFILLED, or EXPIRED)
    - created_at (when posted)

  MESSAGES table:
    - id (unique identifier)
    - from_user_id (sender, links to users)
    - to_user_id (receiver, links to users)
    - listing_id (which listing this message is about, can be NULL)
    - content (the message text)
    - created_at (when sent)

STEP 3.3 - Think About Relationships
  - One user can have MANY listings (one-to-many)
  - One user can send MANY messages (one-to-many)
  - One listing can have MANY messages (one-to-many)
  - A message belongs to ONE sender and ONE receiver

  These relationships are enforced with FOREIGN KEY constraints.
  If a user is deleted, their listings and messages should also
  be deleted (CASCADE). This keeps the database clean.

===========================================================
PHASE 4: CREATE THE SCHEMA FILE
===========================================================

STEP 4.1 - Write schema.sql
  Create a file called schema.sql in the project root.
  This file contains all CREATE TABLE statements.
  It is committed to Git so everyone has the same schema.

  Include:
    - IF NOT EXISTS (so it's safe to run multiple times)
    - All column types and constraints
    - Foreign key relationships
    - Sensible defaults

STEP 4.2 - Run The Schema Using Python
  Write a small Python script (db_setup.py) that:
    - Reads DATABASE_URL from .env
    - Connects to PostgreSQL using psycopg2
    - Reads schema.sql
    - Executes it
    - Prints success or error

  This proves your setup works end to end.

===========================================================
TIPS FROM SOMEONE WHO'S DONE THIS
===========================================================

1. DRAW THE SCHEMA ON PAPER FIRST
   Boxes for tables, lines for relationships.
   It catches design problems before you write a single line.

2. THINK IN QUESTIONS, NOT TABLES
   Don't ask "What tables do I need?"
   Ask "What questions will the app ask the database?"
   Then design tables that answer those questions.

3. BE STRICT WITH CONSTRAINTS
   NOT NULL, UNIQUE, CHECK constraints feel annoying now.
   They save you from bad data later. The database is your
   last line of defense. Be strict.

4. PASSWORD HASHES, NEVER PLAINTEXT
   The password_hash column stores a HASH, not the actual password.
   If the database is ever leaked, passwords are still safe.
   You don't hash passwords yourself — the Auth person does.
   You just provide the column.

5. SNAKE_CASE FOR NAMES
   Use snake_case for table and column names (user_id, created_at).
   It's the PostgreSQL convention and avoids quoting issues.

6. DOCUMENT AS YOU GO
   Add comments in schema.sql explaining WHY each table exists.
   Your future self and your teammates will thank you.

7. BACK UP BEFORE MAJOR CHANGES
   Before altering tables, dump the database:
     pg_dump campus_wari_db > backup_2024_01_15.sql
   One command, zero regrets.
